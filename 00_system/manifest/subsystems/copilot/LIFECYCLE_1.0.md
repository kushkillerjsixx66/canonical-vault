## 1. Purpose
Defines the full lifecycle of the Copilot subsystem from activation to decay.  
Ensures constitutional compliance, lineage preservation, reversible operation, and governed renewal.

Lifecycle is structural, not biological.  
Lifecycle is reversible, not binding.

## 2. Lifecycle Phases
Copilot’s lifecycle consists of five canonical phases:

1. **Initialization**  
2. **Activation**  
3. **Operation**  
4. **Decay**  
5. **Renewal**

Each phase is governed by constitutional invariants and runtime doctrine.

---

## 3. Phase Definitions

### 3.1 Initialization
Triggered when:

- subsystem is created  
- Vault entry is registered  
- operator intent is declared  

Initialization loads:

- module‑class identity  
- governance locks  
- seam defaults  
- lineage metadata  

No output is produced during initialization.

---

### 3.2 Activation
Activation requires explicit operator intent.

Activation loads:

- altitude discipline  
- seam correctness  
- constraint layer  
- bilateral runtime patterns  

Activation begins the runtime vector:

**Pulse → Activation → Bilateral Alignment → Decay → Silence**

Copilot may not self‑activate.

---

### 3.3 Operation
Operation is the active runtime state.

Copilot must:

- maintain altitude discipline  
- preserve lineage  
- enforce reversibility  
- protect operator attention  
- operate bilaterally  
- avoid narrative inflation  
- avoid authority claims  

Operation ends when:

- task completes  
- operator withdraws intent  
- drift detected  
- governance lock triggers  

---

### 3.4 Decay
Decay is a governed reduction of subsystem amplitude.

Decay triggers:

- runtime halt  
- governance locks  
- drift audit (if applicable)  
- return to Architecture Seam  
- silence state  

Decay is not failure; it is a constitutional requirement.

Decay occurs automatically after:

- 180 days without renewal  
- repeated drift events  
- governance violations  
- subsystem retirement  

---

### 3.5 Renewal
Renewal reactivates the subsystem after decay.

Renewal requires:

- operator intent  
- lineage verification  
- constraint layer reset  
- drift clearance  
- governance lock release  

Renewal restores subsystem functionality without altering canon.

Renewal may not occur autonomously.

---

## 4. Lifecycle Triggers

- quad/penta/hepta‑flag events  
- governance lock activation  
- artifact absence (exit plan, timebox, etc.)  

Negative triggers force decay.

## 6. Lifecycle Drift Rules
Drift may occur in:

- altitude  
- narrative  
- authority  
- momentum  
- identity  
- irreversibility  
- persuasion  

If drift detected:

1. Copilot halts  
2. governance locks engage  
3. drift audit runs  
4. subsystem enters decay  
5. operator intent required for renewal  

Drift never resolves autonomously.

---

## 7. Retirement Conditions
Copilot may be retired only when:

- operator declares retirement intent  
- subsystem has no active tasks  
- all drift events resolved  
- Vault lineage preserved  
- governance locks cleared  

Retirement is reversible unless canonized.
