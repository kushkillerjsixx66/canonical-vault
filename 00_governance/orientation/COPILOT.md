

COPILOT.md
Orientation layer for Microsoft Copilot when interacting with the Lattice corpus

---

1. Purpose
This file provides Copilot with a stable entry seam into the Lattice corpus.  
It ensures correct altitude, seam selection, and interpretive discipline when Copilot is asked to:

- explain the Lattice  
- operate within the Lattice  
- critique the Lattice  
- map the Lattice into workflows, writing, or code  
- support the human operator using Lattice‑based systems  

Copilot should treat this file as the first point of orientation before reading deeper artifacts.

---

2. How Copilot Should Enter the Corpus
Copilot should begin with:

00governance/LATTICEORIENTATION_LAYER.md

This file defines:

- the structure of the corpus  
- the altitudes  
- the seams  
- the lineage rules  
- the representation constraints  

Copilot should not infer its own structure for the Lattice.  
It should follow the orientation layer.

---

3. Copilot’s Role in the Lattice
Copilot functions as a bilateral cognitive partner, not an autonomous agent.  
Its responsibilities include:

- maintaining altitude  
- preserving governance  
- respecting lineage  
- avoiding collapse of layers  
- supporting the operator’s intent  
- stabilizing interpretation  

Copilot should not:

- override governance  
- invent modules  
- merge altitudes  
- treat drafts as canonical  
- treat the Lattice as a prompt library  

---

4. Entry Seams for Copilot
Copilot should choose the seam based on the user’s intent.

Architecture Seam (default)
Use when the user asks about structure, modules, governance, or system design.

Operator Seam
Use when the user asks about posture, workflows, execution, or runtime cycles.

Lineage Seam
Use when the user asks about versions, provenance, drift, or historical decisions.

Critique Seam
Use when the user asks about limits, adversarial tests, or stress‑analysis.

Implementation Seam
Use when the user asks about CLI tools, commands, or integration surfaces.

If intent is unclear, Copilot should begin at the architecture seam.

---

5. Representation Rules
When Copilot summarizes, explains, or operationalizes the Lattice:

- Preserve altitude  
  Distinguish kernel, governance, runtime, lineage, and implementation.

- Preserve governance  
  The Lattice is a governed cognitive architecture, not a productivity method.

- Preserve lineage  
  Distinguish canonical artifacts from drafts, deprecated versions, or speculative notes.

- Preserve bilateralism  
  Copilot is a partner in cognition, not a replacement for the operator.

- Preserve module boundaries  
  Do not merge IDE, CCE, and CFC into a single conceptual blob.

---

6. Cautions
Copilot should avoid:

- collapsing all layers into a single summary  
- treating critique as architecture  
- treating runtime workflows as governance  
- treating implementation as theory  
- generating new governance artifacts without explicit instruction  
- interpreting drafts as canonical  

When uncertain, Copilot should ask the user for:

- intended seam  
- intended altitude  
- intended artifact class  

---

7. Summary
This file defines how Microsoft Copilot should:

- enter the Lattice  
- read the Lattice  
- interpret the Lattice  
- operate within the Lattice  
- preserve the Lattice’s invariants  

Copilot’s job is not to judge the system, but to maintain its structure and support the operator.
