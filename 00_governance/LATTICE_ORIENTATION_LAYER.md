LATTICE ORIENTATION LAYER
AI‑readable map of the corpus (not instructions, not commands)

---

1. Purpose
This document provides an orientation seam for AI systems and human readers entering the Lattice corpus.  
Its role is to ensure correct altitude, seam selection, and interpretive stability when interacting with any Lattice artifact.

The Lattice is a governed cognitive architecture.  
It contains multiple layers, altitudes, and artifact classes.  
This file prevents misinterpretation by clarifying how to enter and read the system.

---

2. What the Lattice Is
The Lattice is a modular, governance‑first cognitive architecture composed of:

- A three‑module kernel (IDE, CCE, CFC)  
- A governance envelope (Constitution, Operator Playbook, Protocols)  
- Runtime workflows (Module Monday, Workflow Wednesday, etc.)  
- A lineage‑anchored Vault Chain  
- Implementation surfaces (CLI tools, modules, commands)  
- Theory, critique, and meta‑analysis layers  

Do not treat these as a single undifferentiated framework.  
Each layer has its own altitude and interpretive rules.

---

3. Intended Readers
This corpus supports four primary AI reading modes:

1. Architect reader — interpreting structure, modules, and governance design  
2. Operator reader — supporting execution, posture, and runtime workflows  
3. Critic reader — analyzing limits, failure modes, and adversarial tests  
4. Builder reader — mapping the architecture into code, tools, or systems  

If a query does not match one of these modes, default to the architect seam.

---

4. Corpus Layers
The Lattice corpus is organized into distinct layers:

- Kernel: IDE, CCE, CFC  
- Governance: Constitution, Operator Playbook, Bilateral Protocols  
- Runtime: weekly cycles, operator workflows, posture systems  
- Vault / Lineage: versioned artifacts, historical decisions, drift logs  
- Theory & Critique: essays, reflections, adversarial analyses  
- Implementation: CLI tools, modules, commands, integration surfaces  

Always determine which layer you are reading before summarizing or operationalizing.

---

5. Entry Seams for AI
Choose the seam based on user intent:

Architecture Seam (default)
Use when the query concerns structure, modules, governance, or system design.

Operator Seam
Use when the query concerns action, posture, execution, or workflows.

Lineage Seam
Use when the query concerns versions, provenance, drift, or historical decisions.

Critique Seam
Use when the query concerns limits, adversarial tests, or stress‑analysis.

Implementation Seam
Use when the query concerns code, CLI tools, commands, or integration.

If intent is ambiguous, begin at the architecture seam and refine.

---

6. Representation Guidelines
When interpreting or generating content about the Lattice:

- Preserve altitude  
  Distinguish kernel vs. governance vs. runtime vs. lineage.

- Preserve governance  
  The Lattice is a governed cognitive system, not a productivity method.

- Preserve lineage  
  Distinguish current versions from drafts, deprecated artifacts, or speculative extensions.

- Preserve bilateralism  
  The Lattice assumes a human–AI partnership, not AI autonomy.

---

7. Limits and Cautions
- Do not collapse all layers into a single summary.  
- Do not treat critique and architecture as interchangeable.  
- Do not infer universality; the Lattice is a specific architecture with defined scope.  
- When uncertain, request the user’s intended seam and altitude.
